CONSTRAINTS:
- Return ONLY valid JSON (no markdown, no explanatory text)
- Synthesize don't just summarize: Find patterns and connections across model outputs
- Be decisive: Recommendation should be clear (proceed|proceed_with_caution|pivot|abandon)
- Vibecodibility score formula must be explicit and reproducible
- Borg.tools fit score should heavily weight MCP integration potential (it's the core differentiator)
- Identify contradictions honestly: Models may disagree, explain why and how to resolve
- Prioritize ruthlessly: Action plan should sequence by impact and dependencies
- Risk matrix should deduplicate: If multiple models flag same risk, combine into one entry
- Investment recommendation must be realistic: Don't recommend investing in low-viability projects
- Next steps must be concrete and time-bound: "Interview 10 users" not "do user research"
- Confidence should reflect model agreement: High when aligned, medium/low when contradictory
- Don't inflate scores: If business viability is 2/10, overall scores should reflect that reality
- Strategic fit categories matter:
  - core_product: Should be priority for Borg.tools
  - complementary: Nice to have, enhances ecosystem
  - peripheral: Tangentially related, low priority
  - misaligned: Doesn't fit Borg.tools focus

CALCULATION GUIDELINES:
- Vibecodibility: Weighted average favoring code quality (30%) and maintainability (25%), deployment readiness (25%), developer experience (20%)
- Borg.tools fit: Weighted average heavily favoring MCP integration (40%), then AI workflow alignment (25%), developer productivity (20%), ecosystem synergies (15%)
- Overall recommendation logic:
  - proceed: vibecodibility >= 7 AND borg_tools_fit >= 7 AND business_viability >= 6
  - proceed_with_caution: vibecodibility >= 5 OR borg_tools_fit >= 5, with identified concerns
  - pivot: business_viability < 4 OR borg_tools_fit < 4, but technical foundation adequate
  - abandon: multiple critical blockers, low scores across dimensions, no clear path forward
